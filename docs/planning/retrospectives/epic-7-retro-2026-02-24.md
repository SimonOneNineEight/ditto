## What Could Improve

### 1. User Journey Gaps in Story Planning (Priority: HIGH)

**Issue:** Application attachments bug found in production — edit mode only showed upload dropzone, not existing uploaded files. The story planning didn't consider the "returning user" editing existing data.

**Impact:** Bug shipped to production and was only caught through manual testing (dogfooding).

**Resolution:** Adopt the **five-state UI checklist** for all UI stories during planning:
1. What does the user see the **first time**? (empty state)
2. What does the user see when **creating**?
3. What does the user see when **coming back** to view?
4. What does the user see when **editing** existing data?
5. What does the user see when **something goes wrong**?

### 2. Infrastructure Uncertainty Underestimated (Priority: MEDIUM)

**Issue:** Deploy pipeline (Story 7.2) required fundamental approach change — GHCR pull designed in tech-spec didn't work smoothly, evolved to SCP-based transfer across 3 failed PRs.

**Impact:** More iteration than expected, though the outcome was actually simpler than the original design.

**Resolution:** Infrastructure stories should begin with a **spike** to validate the approach against the real environment before full implementation.

### 3. Configuration Blind Spot (Priority: MEDIUM)

**Issue:** Every story's review flagged configuration/wiring issues: NEXT_PUBLIC_API_URL pointing to Docker-internal hostname (7.1), pnpm test syntax (7.2), endpoint flag handling inconsistency (7.3).

**Impact:** Configuration errors don't surface in unit tests — they only appear when running the full stack in the target environment.

**Resolution:** **Environment validation ladder** — verify config works across local → Docker → production contexts before marking done.

## Key Insights

### 1. User Journey Thinking Prevents UX Gaps
The attachments bug was a planning gap, not a coding mistake. Nobody asked "what does the user see when editing?" The five-state checklist (empty/create/view/edit/error) catches these at story definition time.

### 2. Infrastructure Carries Higher Uncertainty Than Features
Feature stories follow established patterns. Infrastructure stories hit external systems, network configs, and auth flows that can't be fully validated locally. Budget spikes for real-environment validation.

### 3. Configuration Is the Blind Spot
Every story review flagged wiring issues. Unit tests don't catch them. The environment validation ladder (local → Docker → production) closes this gap.

### 4. Closing Deferred Decisions Pays Compound Returns
Infrastructure debt from Epics 5-6 compounded into blockers. Epic 7 resolved all of it in one focused push. Lesson: technical decisions should not drift past one epic.

### 5. Dogfooding Is the Best QA
Simon found the attachments bug by using the product in production. Real usage catches what automated tests miss. Sharing with friends will surface the next round of insights.
