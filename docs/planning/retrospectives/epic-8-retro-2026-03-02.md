## What Could Improve

### 1. Story 8.3 Link Flow Required 3 Iterations (Priority: MEDIUM)

**Issue:** The provider linking flow hit a CSRF wall — the NextAuth signIn callback runs server-side and can't include CSRF tokens for backend API calls. Required 3 approaches before finding the working architecture (server action → URL params → cookie-based client-side).

**Impact:** Extra development time and complexity on what seemed like a straightforward feature.

**Root Cause:** Design work happened after story creation. The interaction between NextAuth server-side callbacks and CSRF-protected endpoints wasn't explored during planning.

**Resolution:** Adopt **draft → design → refine** workflow: draft story first (establishes technical scope), do design work (explores UX and discovers edge cases), then refine story (captures what design revealed).

### 2. Mid-Story Scope Expansion (Priority: LOW)

**Issue:** Provider email display was added during Story 8.3 implementation. Required full-stack changes: migration 000016, model update, repository changes, handler changes, frontend changes.

**Impact:** Small in this case, but demonstrates how design decisions made after story creation expand scope.

**Resolution:** Same as above — the design phase would have identified the need for provider email display before development started.

### 3. Production Deployment Not Done Before Retro (Priority: LOW)

**Issue:** Epic 8 is code-complete but not yet deployed to production. Retro ran before production validation.

**Impact:** Production-specific issues (migration on real data, LinkedIn OAuth callback URLs) haven't been validated yet.

**Resolution:** Going forward, **deploy before retro** to catch production issues while context is fresh.

## Key Insights

### 1. Draft → Design → Refine Prevents Mid-Story Surprises
Story 8.3's CSRF issue and provider_email addition both stemmed from design decisions made after story creation. A draft → design → refine workflow would surface these requirements before development starts.

### 2. Backend-First Ordering Works Well for Full-Stack Features
The strict 8.1 → 8.2 → 8.3 sequence meant each story had stable foundations. No integration surprises or API contract changes mid-epic.

### 3. Established Patterns Accelerate Delivery
Stories 8.1 and 8.2 were fast because they followed existing Handler → Repository → Model patterns exactly. Story 8.3 was slower because the link flow was genuinely novel (NextAuth + CSRF interaction had no existing pattern to follow).

### 4. Deploy Before Retro
Production can reveal issues that dev environments miss — Epic 7's attachments bug proved this. Running the retro before deployment means any production issues won't be captured in the retrospective.
